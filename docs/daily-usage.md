# StoryMaker 日常使用

StoryMaker 的设计目标是：作者审阅创作结果，AI agent 处理底层流程。

## 理想循环

```text
用户：继续写下一章。

StoryMaker Agent：
1. 恢复当前工作流状态。
2. 获取本章需要的上下文。
3. 生成下一工作单元。
4. 运行质量检查。
5. 准备待确认知识更新。
6. 展示正文和质量报告。
7. 等待用户通过或打回。
```

用户看到的是结果，而不是命令链。

## 常用说法

可以使用短句：

```text
继续。
继续写下一章。
显示当前进度。
通过。
通过并继续。
打回：主角选择不可信。
修改，开头钩子更强一点。
把这一章改名为《雨夜来客》。
```

## Agent 职责

Agent 应该：

- 在行动前运行 `storymaker status` 或 `storymaker resume`。
- 如果已有章节等待审阅，不要启动新章节。
- 写作前获取聚焦上下文。
- 每生成一章就停下，请用户审阅。
- 只有用户明确通过后，才提交正文和知识更新。
- 保留被打回的草稿和待确认事实，避免污染正式设定。

Agent 不应该：

- 要求用户手动执行一长串 StoryMaker 命令。
- 把未审阅内容标记为最终正文。
- 静默覆盖故事文件或项目设置。
- 把假设当成正式设定。

## 用户审阅内容

每次章节审阅应包含：

```text
章节正文路径
质量报告路径
简短质量摘要
已知风险或连续性缺口
是否通过的问题
```

示例：

```text
第 0007 章已完成。

正文：outputs/chapters/第 0007 章 雨夜来客.md
报告：reviews/run-...-chapter-0007.md

摘要：
- 连续性：没有发现阻塞冲突。
- 文风：已修正一个模板化表达。
- 知识库：有 3 条待确认事实。

是否通过这一章？
```
